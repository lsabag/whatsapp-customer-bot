שלב ראשון // הוכחת היתכנות

חלק א': השרת המקומי (MVP)

המטרה שלנו כאן היא להגיע לתוצאה מהירה ("וואו" ראשוני) בעזרת כלי פיתוח נגישים. זהו אב-טיפוס שמיועד לדמו בלבד.

חומרה וסביבה

  • ■ מחשב אישי עם Node.js (גרסה 18+): סביבת הריצה של הקוד.
  • ■ חיבור אינטרנט יציב: חובה לתקשורת רציפה עם ה-AI.
  • ■ חשיפת המחשב (Ngrok / Cloudflare Tunnel): יצירת "מנהרה" כדי שווטסאפ תוכל לשלוח הודעות ישירות למחשב הביתי שלך.

גישות ומפתחות

  • ■ מפתח Anthropic API: שימוש במודל קלוד. קריטי לשפה העברית וליכולת שלו להישמע להוראות מערכת קפדניות.
  • ■ מספר ווטסאפ ייעודי: לא המספר האישי שלך! מספר נפרד שנועד רק לסריקת ה-QR ולטסטים.

הקמה והרצה באוויר

> git clone https://...
> npm install
> echo "ANTHROPIC_API_KEY=xxx" > .env
> npm start

1. התקנה: משיכת הקוד והתקנת ספריות.
2. הזרקת ידע: עדכון תיקיית ה-knowledge/ עם המידע הספציפי על העסק (RAG).
3. הרצה: הקלדת הפקודה npm start, סריקת קוד ה-QR שמופיע במסך, והבוט מחובר.

מודעות וסיכונים

שימוש בספריות כמו whatsapp-web.js (שמדמות דפדפן כדי לסרוק QR) הוא אינו רשמי בעיני פייסבוק.

הסכנה: המערכות של מטא עלולות לזהות פעילות רובוטית ולחסום את המספר לצמיתות (Ban). לכן, שיטה זו מתאימה אך ורק לדמו קצר, ואסור לחבר אליה מספר של עסק אמיתי פעיל.

שלב שני // יציבות מערכתית

חלק ב': המעבר לפרודקשן

כשמעבירים את זה מעסק ללקוח שמשלם כסף, אנחנו עוזבים את ה"מחשב בבית" ועוברים למוצר עמיד, יציב ב-100% וחוקי.

WhatsApp Cloud API רשמי

המעבר הקריטי ביותר. התחברות ל-API החוקי של Meta (ישירות או דרך Twilio). אמנם יש עלות לכל הודעה, אך זה מונע חסימות לצמיתות ומאפשר עבודה שקטה ובטוחה.

שרת בענן (Cloud Hosting)

במקום להשאיר את המחשב בבית דולק, מעלים את הקוד לשרת אמיתי (כמו Oracle Cloud, AWS, או Cloudflare Workers) שמבטיח זמינות המערכת 24/7 ללא נפילות אינטרנט מקומיות.

בסיס נתונים מתמשך (Database)

כדי שהבוט לא ישכח על מה הוא מדבר כשהשרת עושה ריסטארט, מחברים אותו למסד נתונים (כמו Postgres או Supabase) לניהול היסטוריית השיחות והזיכרון הארוך של הבוט.

ניהול, משפט והמוצר עצמו

ניהול ואבטחה

  • הפרדת סביבות: הפרדת מפתחות API וטוקנים לכל לקוח בנפרד (Multi-tenancy).
  • לוגים וניטור: מערכות שמתריעות לכם לטלפון בזמן אמת על קריסות (לפני שהלקוח מתלונן).
  • מנגנון הסלמה (Fallback): הבוט מזהה קושי או כעס, עוצר את ה-AI ומעביר שיחה לנציג מיד.

משפטי ועסקי

  • Opt-in: חובה לדרוש הסכמת פרטיות מהלקוח לקבלת הודעות אוטומטיות.
  • SLA: הסכם שירות שמגן עליכם משפטית (פטור מאחריות במקרה של טעות AI).

הממשק ומודל התמחור

הלקוח לא יודע לתכנת. חייבים לתת לו פאנל ניהול פשוט באינטרנט שבו הוא מעדכן את התוכן והמידע של הבוט באופן עצמאי.

מודל רווח: דמי הקמה חד-פעמיים (על בסיס הידע והחיבור) + מנוי חודשי קבוע (Retainer) על שרתים ותמיכה.

תוספות קריטיות לפרימיום

כדי להפוך את זה מסתם מדריך למוצר ברמת פרימיום אמיתית, הוספתי עוד עקרונות עבודה של מפתחים בכירים:

🪂

1. מנגנון "נחיתה רכה"

הדבר הכי חשוב למנוע הלוצינציות (המצאות) של הבינה המלאכותית. תמיד להגדיר לבוט בקוד פקודה קשוחה:

"אם אתה לא יודע את התשובה בוודאות לפי המסמכים - אל תמציא מידע בשום אופן. במקום זאת, הצע ללקוח להמתין לעזרה מנציג אנושי."

👁️

2. בקרה אנושית (Human In Loop)

הפיצ'ר שהכי מרגיע לקוחות ובעלי עסקים.

חובה לספק לבעל העסק ממשק שבו הוא רואה את שיחות הבוט מתנהלות בזמן אמת. המערכת חייבת לאפשר לו להתערב בשיחה בכל רגע נתון, כשברגע ההתערבות - הבוט מושתק (Muted) אוטומטית.

3. ולידציה למידע (QA)

לעולם לא מחברים בוט ללקוחות אמיתיים בלי בדיקה מקיפה.

לפני שהבוט מתחיל לעבוד על המידע של הלקוח, מריצים סקריפט בדיקה אוטומטי (QA) שזורק על הבוט 10 שאלות מאתגרות ונפוצות. רק לאחר שמוודאים שהתשובות הגיוניות ומדויקות - המערכת עולה לאוויר.
